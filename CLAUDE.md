## Repository Structure

This repository has three layers managed independently:

```
.claude/          → Claude Code management (plans, reports, commands, refs)
paper/            → LaTeX paper source (managed by Overleaf Git)
project/          → Engineering code (managed by GitHub Git)
logs/             → Experiment logs (not tracked by git)
```

### `.claude/` — Development Management

| Path | Purpose |
|------|---------|
| `STATUS.md` | **Read this first every session.** Current project state, recent progress, next steps. |
| `plans/vN.md` | Development plan for iteration N. Input for `/dev`. |
| `reports/vN.md` | Completion report for iteration N. Generated by `/report`. |
| `commands/` | Custom slash commands (`/dev`, `/report`, `/sync-paper`). |
| `refs/` | Read-only reference materials. See `refs/README.md` for index. |

### `paper/` — LaTeX Paper (Overleaf Git)

Split by chapter. `main.tex` only contains `\input{}` directives — no prose content. Exmaple:

| Path | Content |
|------|---------|
| `chapters/01_introduction.tex` | Introduction |
| `chapters/02_related_work.tex` | Related Work |
| `chapters/03_method.tex` | Method |
| `chapters/04_experiments.tex` | Experiments |
| `chapters/05_results.tex` | Results & Analysis |
| `chapters/06_conclusion.tex` | Conclusion |
| `main.tex` | Entrance for the paper content |
| `tables/` | Standalone table files (e.g., `main_results.tex`, `ablation.tex`) |
| `figs/` | Figures |
| `references.bib` | Bibliography |

**Important**: The `paper/` directory is managed by Overleaf's Git remote, not the main GitHub repo. Do not run `git` commands inside `paper/` unless explicitly asked.

### `project/` — Engineering Code (GitHub Git)

| Path | Purpose |
|------|---------|
| `src/` | Core source code (models, environments, utilities) |
| `scripts/` | Experiment launchers and analysis scripts |
| `configs/` | YAML experiment configurations |
| `tests/` | Unit and integration tests |

### `logs/` — Experiment Logs

Not tracked by git. Each run gets its own subdirectory (e.g., `logs/run_20250615_ablation_lr/`). May be referenced by both code and paper.


## Python Environment

Use `uv` for all environment and package management. **Never use `pip install` directly.**

```bash
# Virtual environment (do NOT recreate)
# Created with: uv venv --python=3.12

# Activate before ANY Python execution
source .venv/bin/activate

# Install packages
uv pip install <package>

# Install from requirements
uv pip install -r requirements.txt
```

**Rules**:
- Always activate the venv before running Python scripts, tests, or installing packages.
- If a required package is missing, install it with `uv pip install` and note it in the report.
- Do not modify `pyproject.toml` or `requirements.txt` without explicit instruction.

## Code Conventions

### Python Style
- Type hints on all function signatures.
- Docstrings (Google style) on all public functions and classes.
- Use `pathlib.Path` over `os.path`.
- Use `logging` module, not `print()`, for status messages in library code.
  `print()` is acceptable in scripts.

### Configuration
- Experiment configs live in `project/configs/` as YAML files.
- All experiments are launched via `project/scripts/launch_experiment.py`.
- Hardcoded hyperparameters in source code are not acceptable — use config files.

### Testing
- Tests live in `project/tests/`, mirroring the `src/` structure.
- Run tests: `source .venv/bin/activate && cd project && python -m pytest tests/`
- Run a specific test: `python -m pytest tests/test_model.py -v`
- Always run affected tests after making changes. If tests fail, fix before finishing.

### LaTeX Style
- One sentence per line in `.tex` files (makes git diffs cleaner).
- Tables are defined in `paper/tables/` as separate files, included via `\input{tables/...}`.
- Use `\label{}` and `\ref{}` / `\cref{}` for all cross-references.
- Mark uncertain or draft content with `% TODO: ...` comments.


## Git Conventions

### Commit Message Prefixes

| Prefix | Scope | Example |
|--------|-------|---------|
| `[project]` | Engineering code changes | `[project] add attention module to encoder` |
| `[paper]` | Paper content changes | `[paper] update results table with ablation` |
| `[claude]` | Plans, reports, status | `[claude] add v3 report` |
| `[config]` | Configuration changes | `[config] add ablation config for lr sweep` |
| `[fix]` | Bug fixes | `[fix] resolve memory leak in data loader` |

### What NOT to Commit
- `logs/` — experiment outputs (large, ephemeral)
- `.venv/` — virtual environment
- `__pycache__/`, `*.pyc` — Python cache
- `paper/*.aux`, `paper/*.log`, etc. — LaTeX build artifacts

## Constraints and Guardrails

- **Never delete or overwrite** files in `.claude/refs/`. They are read-only references.
- **Never run `git push`** without explicit human instruction.
- **Never modify `paper/` files** unless the task specifically involves the paper (e.g., `/sync-paper`).
- **Always run tests** after modifying code in `project/src/`.
- **Preserve existing code style** — match the patterns already in the codebase rather than introducing new conventions.
- When in doubt about a design decision during `/dev`, **document the decision and rationale** rather than asking — the human will review in the report.
