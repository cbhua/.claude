## Python Environment

Use `uv` for all environment and package management. **Never use `pip install` directly.**

```bash
# Virtual environment (do NOT recreate)
# Created with: uv venv --python=3.12

# Activate before ANY Python execution
source .venv/bin/activate

# Install packages
uv pip install <package>

# Install from requirements
uv pip install -r requirements.txt
```

**Rules**:
- Always activate the venv before running Python scripts, tests, or installing packages.
- If a required package is missing, install it with `uv pip install` and note it in the report.
- Do not modify `pyproject.toml` or `requirements.txt` without explicit instruction.


## Development Workflow

### How This Project Iterates

1. Plans are prepared in `.claude/plans/vN.md` (generated by `/plan` command)
2. Execute with `/dev vN` — Claude Code reads the plan and implements it, and finally generates a report `.claude/reports/vN.md` and updates `STATUS.md`
3. Human reviews, then starts the next planning cycle

### Session Startup Checklist

**At the beginning of every session, read these files in order**:
1. This file (`CLAUDE.md`) — project structure and conventions
2. `.claude/STATUS.md` — current progress, recent changes, known issues
3. The relevant plan in `.claude/plans/` if executing `/dev`

### Slash Commands

| Command | Usage | Description |

## Code Conventions

### Python Style
- Type hints on all function signatures.
- Docstrings (Google style) on all public functions and classes.
- Use `pathlib.Path` over `os.path`.
- Use `logging` module, not `print()`, for status messages in library code.
  `print()` is acceptable in scripts.

### Testing
- Tests live in `tests/`, mirroring the `src/` structure.
- Run tests: `source .venv/bin/activate && python -m pytest tests/`
- Run a specific test: `python -m pytest tests/test_model.py -v`
- Always run affected tests after making changes. If tests fail, fix before finishing.

## Constraints and Guardrails

- **Never delete or overwrite** files in `.claude/refs/`. They are read-only references.
- **Never run `git push`** without explicit human instruction.
- **Always run tests** after modifying code in `src/`.
- **Preserve existing code style** — match the patterns already in the codebase rather than introducing new conventions.
- When in doubt about a design decision during `/dev`, **document the decision and rationale** rather than asking — the human will review in the report.
